# Exercisable Verification Interface — Reproducible Verification, Not Validator Authority

## Status

Open architectural design note.

This document formalizes the interface layer required to make A-DAP verification exercisable by non-technical affected parties without turning the verifier into a new authority.

The central correction is:

The validator must not become an attestation authority.

It must be an open reproducibility layer.

A-DAP should not replace operator self-validation with validator self-validation.

The purpose of the interface is to let affected parties, auditors, regulators, courts, civil society, and independent reviewers reproduce the same deterministic verification result from the same decision receipt.

---

## 1. Core Claim

A-DAP does not need a new authority that validates decision receipts.

It needs a reproducible verification interface that makes verification exercisable.

The difference is critical.

Unsafe framing:

“The validator confirms that this receipt is valid.”

Safer framing:

“Running this open tool, at this version, over this receipt, with these inputs, produces this deterministic result.”

The force of the result comes from reproducibility, not from the authority of the validator.

---

## 2. Why This Matters

A-DAP exists because systems should not validate themselves.

If the verification interface produces a human-readable report that functions as an unquestioned certificate, the same failure mode returns.

The system would move from:

“Trust the operator.”

to:

“Trust the validator.”

That collapses the independence A-DAP is trying to preserve.

A report generated by a validator is not proof by itself.

It is useful only if another party can independently reproduce the same result.

Therefore, the interface must separate:

1. deterministic computation,
2. human interpretation,
3. institutional judgment.

Those three layers must not be collapsed.

---

## 3. Design Principle

The validator must compute.

It must not rule.

It may expose deterministic findings such as:

- hash match,
- signature status,
- timestamp presence,
- Merkle proof status,
- schema declared,
- required fields under that schema,
- present fields,
- missing fields,
- operator dependencies,
- collapsed NDC estimate,
- reproduction command,
- tool version,
- input hash.

It may also provide human-readable interpretation.

But interpretation must be clearly marked as interpretation.

The interface must never present interpretation as proof.

---

## 4. What the Interface Should Be Called

The phrase “universal validator” is risky if it implies broad authority.

Safer names include:

- A-DAP Decision Receipt Reproducibility Tool
- A-DAP Receipt Reproduction Interface
- Universal Decision Receipt Reproducibility Layer
- Exercisable Verification Interface
- Reproducible Decision Receipt Checker

A possible public-facing name is:

A-DAP Decision Receipt Validator

But the documentation must clarify:

It validates reproducibility conditions.

It does not validate the truth, fairness, legality, or correctness of the decision.

---

## 5. What It Must Not Claim

The interface must not claim:

- the decision was correct,
- the decision was fair,
- the decision was lawful,
- the envelope was complete in an absolute sense,
- the institution is accountable,
- the affected party has won a contestation,
- the verifier is an authority,
- the result is final.

The interface can only show what follows from the receipt, the declared schema, the available proofs, the tool version, and the reproducible verification procedure.

---

## 6. Correct Output Philosophy

A weak validator says:

“Valid.”

A safer reproducibility interface says:

“Using tool version X, receipt hash Y, schema Z, and verification procedure W, the following deterministic checks return these results.”

This distinction prevents the interface from becoming a compliance badge.

The output should be reproducible, inspectable, and challengeable.

---

## 7. Two-Layer Architecture

The interface should have two strictly separated layers.

### 7.1 Deterministic Layer

This layer performs reproducible checks.

Examples:

- parse receipt,
- compute hash,
- verify signature,
- verify timestamp evidence,
- verify Merkle inclusion proof,
- check canonicalization,
- compare fields against declared schema,
- identify missing schema-required fields,
- detect operator-mediated dependencies,
- compute conservative structural NDC where possible,
- produce machine-readable output.

This layer must be:

- open-source,
- versioned,
- deterministic,
- executable by third parties,
- reproducible offline where possible,
- auditable,
- accompanied by input hashes and tool hashes.

### 7.2 Interpretive Layer

This layer translates deterministic output into human language.

Examples:

“Hash verification passed.”

“Timestamp evidence is present, but the timestamp authority is not independently checked by this run.”

“The declared schema requires policy_version, but this field is absent.”

“The receipt can be parsed, but verification depends on an operator-controlled URL.”

This layer must be clearly labeled as interpretation.

It must not become an authority statement.

---

## 8. Reproducibility Package, Not Legal Report

The interface should not produce a “legal report” or “official attestation.”

It should produce a reproducibility package.

Recommended label:

Reproducible Verification Output

Or:

Decision Receipt Reproducibility Package

The package should include:

1. receipt analyzed,
2. receipt hash,
3. tool name,
4. tool version,
5. tool hash or release reference,
6. command or verification method,
7. declared schema,
8. deterministic check results,
9. missing fields under declared schema,
10. dependency findings,
11. optional human-readable interpretation,
12. files or data required for reproduction,
13. limitations of the verification.

The key sentence:

This output is not an authority statement. It is a reproducibility package.

---

## 9. Example Human-Facing Output

A non-technical affected person should see language like:

Verification result:

- Receipt parsed successfully.
- The receipt hash matches the submitted content.
- A signature is present and verifies under the declared public key.
- Timestamp evidence is present.
- The declared schema is credit-denial-v0.1.
- The declared schema requires threshold_commitment.
- threshold_commitment is missing.
- The verification path depends on an operator-controlled portal.
- Conservative dependency analysis indicates limited independence.

Interpretation:

This receipt contains some reproducible evidence, but it does not expose all fields required by the declared schema. The verification path also appears to depend on infrastructure controlled by the operator.

Possible next step:

You may use this reproducibility package to ask the operator, auditor, regulator, or legal representative to explain the missing fields or provide an independently reconstructible receipt.

---

## 10. What the Output Should Avoid

The output should not say:

“This decision is invalid.”

“This institution violated the law.”

“This person was unfairly treated.”

“This receipt proves accountability.”

“This case is won.”

“This verifier certifies the claim.”

Instead, it should say:

“The following deterministic checks passed.”

“The following deterministic checks failed.”

“The following fields are missing under the declared schema.”

“The following dependencies limit independent verification.”

“The following interpretation may support further contestation.”

---

## 11. Scope Inspector Correction

The Scope Inspector is dangerous if it silently defines what a complete envelope should contain.

It must not invent completeness standards.

It must not decide, on its own, what fields are necessary for a domain.

It must only compare the receipt against a declared public schema.

Correct framing:

“The declared schema requires fields A, B, and C. Field B is missing.”

Unsafe framing:

“This envelope is insufficient for credit decisions.”

The second claim is normative unless grounded in an external schema, regulation, contract, public standard, or institutional rule.

Therefore, the Scope Inspector must be schema-bound.

---

## 12. Schema Governance

Schemas should be public, versioned, and challengeable.

Each schema should define:

- domain,
- use case,
- required fields,
- optional fields,
- forbidden omissions,
- reconstruction instructions,
- privacy constraints,
- minimum evidence requirements,
- known limitations,
- version history,
- schema authority or maintainer,
- review status.

Possible schema sources:

- public regulator,
- court rule,
- standards body,
- independent audit consortium,
- civil society working group,
- sector-specific governance body,
- contractual framework,
- open community process.

If the schema is operator-defined, the interface must disclose that.

Example:

“Warning: this schema was defined by the same operator whose decision is being verified.”

---

## 13. Independence Analyzer

The Independence Analyzer is the heart of the interface.

It should detect when apparent verification paths depend on the same authority.

Examples:

- receipt hosted only on operator portal,
- verifier calls operator API,
- schema controlled by operator,
- timestamp claim not externally verifiable,
- reconstruction requires operator credentials,
- audit dashboard controlled by operator,
- proof delivery depends on operator account access.

The interface should not merely say whether proofs exist.

It should ask who controls access to them.

This operationalizes the Envelope Bottleneck and NDC reasoning.

---

## 14. Client-Side Verification Is Not Independence

Client-side verification can reduce dependence on an operator-controlled portal at the moment of execution.

This is useful.

It may reduce friction for affected parties and increase the probability that verification is exercised.

In ADAP-EXP-003 terms, client-side execution may help increase e_i.

But client-side execution does not, by itself, increase NDC.

NDC is not a measure of where computation runs.

NDC is a measure of independent custody and verification paths after dependency collapse.

The relevant distinction is:

- custody vectors,
- verifier distribution vectors,
- and schema governance vectors.

These should not be collapsed into a single category.

### 14.1 Custody Vectors

Custody vectors affect the decision receipt, timestamp, anchoring, or authoritative proof path.

They can enter the NDC/min-cut analysis directly.

Examples include:

- operator-controlled timestamp evidence,
- operator-controlled external anchor access,
- operator-controlled receipt delivery,
- operator-controlled public key directory,
- operator-controlled custody log,
- operator-controlled Merkle root publication,
- operator-controlled proof availability.

If these vectors are controlled by the same operator whose decision is being contested, apparent verification independence may collapse.

A timestamp or external anchor only improves independence if it is reachable and verifiable outside the operator’s custody.

An external anchor whose only accessible proof passes through the operator’s portal does not meaningfully increase NDC.

It reproduces the Envelope Bottleneck in a new location.

### 14.2 Verifier Distribution Vectors

Verifier distribution vectors concern the code used to perform verification.

They are supply-chain risks.

They do not automatically alter custody of the decision receipt or external anchor, but they can compromise the reliability of the verification process.

Examples include:

- operator-controlled JavaScript bundle,
- operator-controlled mobile app,
- unsigned verifier release,
- non-reproducible build,
- unverifiable binary,
- mutable hosted verifier,
- dependency injection through a controlled web interface.

These risks should be handled through:

- open-source code,
- signed releases,
- reproducible builds,
- subresource integrity where applicable,
- tool hashes in verification outputs,
- offline execution where possible,
- independent mirrors,
- and multiple independently runnable implementations.

A compromised verifier can produce false local output.

But that is a verifier supply-chain failure, not the same thing as custody independence over the receipt or anchor.

The two risks must be analyzed separately.

### 14.3 Schema Governance Vectors

Schema control is a scope-governance problem.

A schema defines what the envelope is expected to contain.

If the operator controls the schema, it may omit decisive fields while still producing formally valid receipts.

This does not always mean the timestamp or receipt custody has collapsed.

It means the completeness standard may be captured.

Therefore, schema governance should be disclosed separately.

The interface should distinguish between:

- public or regulator-defined schema,
- independent standards-body schema,
- civil-society or audit-consortium schema,
- contractual schema,
- community-maintained schema,
- and operator-defined schema.

If a schema is operator-defined, the interface should explicitly warn:

“Warning: this schema was defined by the same operator whose decision is being verified.”

The Scope Inspector must not silently convert an operator-defined schema into an independent completeness standard.

### 14.4 Correct Question

The relevant question is not:

“Does verification run client-side?”

The relevant questions are:

- Is the verification procedure deterministic?
- Is the source code public?
- Is the executed artifact reproducibly built from that source?
- Is the artifact signed or hash-pinned?
- Can another party run the same tool over the same receipt?
- Is the timestamp or anchor reachable outside the operator’s custody?
- Can the result be reproduced without relying on the operator’s portal?
- Who controls the schema used to assess missing fields?

Client-side execution is useful for usability and exercise.

It is not sufficient for independence.

The evidentiary force comes from reproducibility, independent custody, external anchoring where applicable, and explicit schema governance.

Do not confuse local execution with independent verification.

---

## 15. Exercise Reduction

ADAP-EXP-003 shows that verification paths are not useful operationally unless they are exercised.

The purpose of the interface is to reduce the cost of exercise.

For a non-technical person, the workflow should be:

1. upload receipt,
2. receive reproducible verification output,
3. understand missing evidence or dependency issues,
4. download a reproducibility package,
5. send it to an auditor, regulator, lawyer, court, ombudsman, public defender, or civil society reviewer.

The interface increases exercise probability not by creating authority, but by reducing friction.

---

## 16. Three User Modes

The same interface should support three modes.

### 16.1 Affected Person Mode

Goal:

Make verification understandable.

Shows:

- plain language findings,
- missing fields,
- dependency warnings,
- next steps,
- downloadable reproducibility package.

Does not show technical details first.

### 16.2 Institutional Reviewer Mode

Goal:

Support auditors, regulators, ombudsmen, lawyers, courts, and public defenders.

Shows:

- receipt metadata,
- schema comparison,
- deterministic check table,
- dependency analysis,
- tool version,
- reproduction instructions,
- limitations.

### 16.3 Technical Reviewer Mode

Goal:

Allow full reconstruction.

Shows:

- raw envelope,
- canonicalized object,
- hashes,
- signatures,
- timestamp evidence,
- Merkle proofs,
- schema files,
- verifier logs,
- command line reproduction,
- dependency graph.

---

## 17. Output Categories

The interface should avoid simple valid or invalid labels.

It should use categories such as:

- parsed,
- not parsed,
- hash matched,
- hash mismatch,
- signature verified,
- signature missing,
- timestamp evidence present,
- timestamp not verified,
- schema declared,
- schema missing,
- required field present,
- required field missing,
- operator dependency detected,
- external anchor detected,
- reconstruction incomplete,
- reproduction package generated.

These are more precise than a single validity label.

---

## 18. Human Summary Format

A good summary for a non-technical person might be:

Summary:

This receipt can be read by the tool.

Some technical checks passed.

However, the receipt is missing fields required by the declared schema, and the verification path appears to depend on the same operator that made the decision.

This does not prove the decision was wrong.

It means the receipt may be insufficient for independent contestation without additional information.

Reproducibility:

Another person can reproduce this result by running tool version X on receipt hash Y using schema Z.

---

## 20. Avoiding Validator Capture

A verifier can itself become captured.

To reduce this risk:

- tool source code should be public,
- releases should be signed,
- output should include tool version and hash,
- results should be reproducible offline where possible,
- schemas should be public,
- operator-defined schemas should be labeled,
- interpretive text should be separable from deterministic output,
- no single hosted instance should be treated as authoritative.

The hosted interface is convenience.

The reproducible procedure is the evidence layer.

---

## 21. Institutional Deployment

There can be multiple deployments of the interface:

- public web interface,
- regulator-hosted interface,
- civil society-hosted interface,
- court or ombudsman interface,
- command-line verifier,
- offline desktop tool,
- browser extension,
- mobile app,
- API for institutional workflows.

No deployment should be the only authority.

The same receipt should produce the same deterministic output across deployments.

---

## 22. Business Model Without Authority Capture

A service may be built around the interface without becoming the authority.

It may charge for:

- usability,
- integration,
- hosting,
- institutional workflows,
- accessibility,
- storage,
- translation of results,
- training,
- support,
- schema management,
- bulk verification,
- regulator dashboards.

It must not sell:

- unquestionable certification,
- proprietary-only verification,
- closed validator authority,
- non-reproducible reports,
- hidden scoring of contestability,
- unverifiable legal conclusions.

The business value should be facilitation of reproducibility, not ownership of truth.

---

## 23. Safe Claim

A safe claim is:

The Exercisable Verification Interface helps non-technical users generate a reproducible verification output from a decision receipt, while separating deterministic checks from human interpretation.

It does not certify that the decision was correct, fair, lawful, or accountable.

---

## 24. Unsafe Claim

An unsafe claim is:

The validator proves whether an automated decision was valid.

This is false.

It recreates the authority problem A-DAP was designed to avoid.

---

## 25. Relation to A-DAP

A-DAP creates decision envelopes.

ADAP-EXP-003 shows that verification must be exercised.

Adoption Capture Risk shows that formal adoption can become compliance theater.

The Exercisable Verification Interface connects these three layers.

It makes verification easier to exercise without turning the interface into a new authority.

---

## 26. Summary

A-DAP should not create a validator that people must trust.

It should create a reproducibility layer that people can run, inspect, compare, and challenge.

The validator does not prove truth.

The validator does not certify fairness.

The validator does not replace institutions.

The validator produces reproducible verification output.

Client-side execution can reduce friction.

But local execution is not independent verification by itself.

The core principle is:

Do not trust the validator.

Reproduce the result.
